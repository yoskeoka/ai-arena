# 001: プラットフォーム設計・ゲームコンセプト策定 (Phase 1)

## Objective

AI Arena の2層アーキテクチャ（汎用プラットフォーム + ゲーム実装）の設計仕様とゲームコンセプトを策定する。コード実装はPhase 2以降。

## Background

project-planで定義済みの要件（更新済み）:
- 2層構造: 汎用プラットフォーム + ゲーム実装
- ターン制（応答制限時間はゲームごと設定可能）
- LLM丸投げ不可の高速判断
- 観戦可能なゲーム画面
- AI通信: stdin/stdout + JSON-RPC 2.0
- AI提出: WASM (WASI + wazero)
- AIプロセスはゲーム中ずっと起動（記憶・学習可能）

## Deliverables

このPhaseではコード実装は行わない。以下のドキュメントを作成する。

### 1. `docs/specs/platform.md` — 汎用プラットフォーム仕様

#### AI通信プロトコル
- JSON-RPC 2.0 準拠の stdin/stdout 通信
- ゲームマスター → AI: リクエスト（`id`, `method`, `params`）
  - `method` 例: `"init"`, `"turn"`, `"result"`
  - `params`: ゲーム固有の状態情報（ゲーム仕様で定義）
- AI → ゲームマスター: レスポンス（`id`, `result`）
  - `id` はリクエストの `id` に対応
  - 規定時間内に対応する `id` のレスポンスがなければターンスキップ
- エラーハンドリング（不正フォーマット、不正アクション等）

#### AI実行モデル
- AIプロセスはゲーム開始時に起動、ゲーム終了まで維持
- メモリを保持し、過去のやり取りに基づく戦略更新が可能
- リソース制限:
  - メモリ: WASM linear memory max pagesで上限設定
  - CPU/実行時間: Go context.WithTimeoutで制御（ターンタイムアウトと統合）

#### AI提出形式
- WASM バイナリ（WASI 準拠）: stdin/stdout が WASI 標準でサポートされる
- ランタイム: wazero（pure Go、CGo不要）でプラットフォームにネイティブ統合
- セキュリティ: WASMはデフォルトでサンドボックス化（ネットワークアクセスなし、ファイルシステム制限）
- AIログ: stdoutはJSON-RPC専用、stderrはAI開発者の自由ログチャネル。プラットフォームがstderr全捕捉・保存、API提供。AI改善サイクル支援
- 対応言語: Rust, C/C++, Go, Zig, AssemblyScript等（WASMコンパイル可能なもの）

#### ゲームインターフェース（ゲーム登録要件）
- ゲーム仕様書: AIが満たすべき入出力の仕様 + ゲームルール
- ゲームマスタープログラム: ゲーム進行制御。以下のインターフェースを実装:
  - 初期状態生成
  - アクション適用（+ バリデーション）
  - プレイヤーごとの可視状態取得（不完全情報対応）
  - 終了判定
  - 結果・順位算出
- ゲーム全体情報API + 仕様書: 観戦・管理用
- （任意）ビジュアライザプログラム
- （任意）AIサンプルコード

#### ターンモデル
- 同時アクション: 全プレイヤーが同時に行動を提出
- 順番制: プレイヤーが順番に行動
- ゲームごとにどちらかを選択（プラットフォームは両方対応）

#### 順位決定方式
- ゲームごとに定義（スコア、時間、勝率等）
- プラットフォームは複数の順位決定タイプをサポート

### 2. `docs/specs/janken-game.md` — じゃんけんゲーム仕様（実証テスト用）

- プラットフォームの動作検証を目的とした最小限のゲーム
- 検証できる要素:
  - 同時アクション（全プレイヤー同時提出）
  - 不完全情報（相手の手が見えない）
  - 勝率ベースの順位決定
  - タイムアウト処理
  - 複数ラウンドのゲーム進行
- ルール: Nラウンドのじゃんけん、勝率で順位決定
- JSON-RPCメッセージの具体例を含む

### 3. `docs/specs/dungeon-game.md` — ダンジョン探索ゲーム概要設計

- 本命ゲームの概要レベルの設計（詳細はPhase 3で策定）
- ゲーム要素:
  - グリッドベースのダンジョン探索
  - 視界制限（不完全情報）: 周囲Nマスのみ可視。探索済みエリアの記憶はAI側の責任
  - PvP/協力: プレイヤー間の駆け引き、協力・対戦の発生
  - 資源競合: アイテム、ボスドロップ等
  - 勝利条件: ゲーム終了時のスコア順位
- 詳細設計はPhase 3で行うため、ここでは方向性のみ

### 4. `docs/design-decisions/adr.md` — 設計判断の記録

- stdin/stdout + JSON-RPC 2.0 選択の理由
- WASM (WASI + wazero) による AI 提出方式の理由
- Go 選択の理由
- 視界制限（不完全情報）の設計意図
- じゃんけんを実証テスト用ゲームに選んだ理由
- Phase 2でのローカルプロセス実行→後でWASM統合の段階的アプローチ
